Getting Started Skills Architecture FAQ

Architecture

Caduceus runs inside Hermes. There's no separate server, no background process, no daemon. The orchestrator skill is just a Hermes skill that uses Hermes tools.

Layers

You — the user

Talk to Hermes. Tell the orchestrator what you need. Get results back. No new interface to learn.

Caduceus Orchestrator (Hermes skill)

The foreman. Spawns agents, tracks tasks, manages retries and approvals. Uses Hermes tools (delegate_task, terminal, read_file, write_file).

Theoi Agents (Hermes skills)

Engineer, Researcher, Writer, Monitor. Each does one thing well. Read SPEC.md, write outputs to QMD, update the coordination log.

SQLite ~/.hermes/caduceus.db

Task state: pending, running, completed, failed. Who is working on what. Execution history. Trigger definitions. Approval queue.

QMD Collections ~/.hermes/caduceus/

Shared knowledge: SPECs, research reports, coordination log, learnings. Agents read and write here. Human-editable. Semantic searchable.

File layout

~/.hermes/
├── skills/                          # Caduceus skills
│   ├── caduceus-orchestrator/      # The foreman
│   ├── caduceus-engineer/          # Builder
│   ├── caduceus-researcher/        # Analyst
│   ├── caduceus-writer/            # Copywriter
│   ├── caduceus-monitor/           # Watcher
│   ├── caduceus-themis/            # Onboarder
│   └── caduceus-kairos/            # Experimenter
├── caduceus/                        # QMD collections
│   ├── agora/                      # Cross-project memory
│   │   ├── coordination/           # task-log.md, blockers.md
│   │   ├── learnings/              # what-worked.md, what-failed.md
│   │   └── decisions/             # Decision records
│   ├── projects/                   # Per-project specs + artifacts
│   └── agents/                    # Per-agent context + state
└── caduceus.db                     # SQLite: tasks, agents, executions,
                                    #           triggers, approvals

Why SQLite + QMD?

SQLite for operational state — who is doing what, right now. ACID guarantees, concurrent write safety, fast lookups by ID.

QMD for knowledge and context — why, what we've learned, what's the bigger picture. Semantic search, human-readable, agent-editable.

They don't overlap. SQLite is the orchestrator's domain (task state). QMD is the agents' domain (shared memory).

Coordination protocol

Agents don't send messages to each other. They write to the QMD coordination log:

## 2026-03-19

- [orchestrator] Received task: build-landing-page (id: task-abc123)
- [engineer] Started: build landing page
- [engineer] Completed: build landing page → Output: projects/ugc/landing.md
- [writer] Started: write landing page copy (depends on: engineer)

The orchestrator reads this to know what's done, what's running, what's blocked. No direct messaging. No message bus. Just a shared log in QMD.

Threat model

Caduceus runs inside Hermes, which means agents can run arbitrary code. Defense in depth:

  • Threat scanning — All prompts scanned for injection patterns before agents run. Invisible unicode, prompt injection, exfiltration, dangerous shell commands.
  • Draft-and-flag — Agents must propose irreversible actions (delete, drop, auth changes). Human approval required before execution.
  • Bounded retries — Tasks retry up to N times (default: 3) before being marked failed. No infinite loops.
  • Depth limit — Sub-agent spawning is depth-limited. Agents can't spawn agents that spawn agents indefinitely.
